01 — BACKGROUND & CHALLENGES

背景と課題

01
深夜の漏水・緊急連絡
夜間・休日の漏水・設備故障連絡が全問い合わせの約18%を占める。電話対応が担当者個人の携帯に依存しており、対応記録が残らない。初動の遅れがクレームに発展するリスクが高い。
02
退去・更新手続きへの質問
「退去通知はいつまでに？」「更新料はいくら？」といった定型の問い合わせが全体の40%超。同じ質問へ毎回個別に返信しており、スタッフの実質的な付加価値が出ない繰り返し業務になっている。
03
オーナー向け報告の工数
物件ごとの入退去状況・修繕履歴・賃料収支をオーナーに月次報告するのに、スタッフが手作業でデータを集約している。200〜500戸規模では月30〜40時間が報告書作成だけで費やされている。
仮説（H-A006）：入居者からの問い合わせ70%をAIが自動対応することでCS工数を半減できる。同時に緊急案件の自動検知で初動対応時間を現状比50%短縮する。
SOLUTION APPROACH
→
LINE Bot経由でAIが24時間一次対応——入居者が使い慣れているLINEを窓口にすることで導入摩擦をゼロにする。FAQ・物件規約・手続きガイドをClaudeに与え、定型問い合わせを即時自動回答する。
→
緊急案件の自動検知とエスカレーション——「水漏れ」「ガス漏れ」「鍵紛失」等のキーワードとAIの感情分析を組み合わせて緊急度を自動分類。担当者のSlack/SMSに即時通知し、対応記録をシステムに自動保存する。
→
オーナーレポートの自動生成——管理DB（空室状況・入退去・修繕履歴・賃料収支）から月次レポートをClaudeが自動作成してオーナーにメール配信。手作業30〜40時間を1クリックに圧縮する。
02 — FUNCTIONAL REQUIREMENTS

機能要件

A. 入居者チャット機能（LINE Bot）

機能 詳細仕様 優先度 フェーズ

入居者本人確認 LINEのuser_idと管理DBの部屋番号を紐づけ登録。初回メッセージ時に部屋番号・姓名で認証。認証済みユーザーのみAI応答を有効化 HIGH Phase 1

B. AI自動応答モジュール（Claude API統合）

機能 詳細仕様 優先度 フェーズ
FAQ自動回答（70%カバー目標） 退去通知期限・更新料・駐車場・ゴミ出しルール・設備操作法など100〜200件のFAQをVector DBで管理。意味的類似検索で最適回答を生成 HIGH Phase 1
問い合わせカテゴリ自動分類 緊急 / 退去 / 更新 / 設備 / 騒音 / その他 の6カテゴリをClaudeが自動分類。カテゴリに応じて回答テンプレート・エスカレーション先を切り替え HIGH Phase 1
物件情報・規約の参照 物件ごとの賃貸借契約書・管理規約・建物仕様書をRAGで参照。物件Aと物件Bで異なるルールを正確に使い分けて回答 HIGH Phase 1
回答信頼度スコアリング AIが自信度スコア（0〜1）を計算。0.7未満の場合は「担当者に確認します」と返してエスカレーションを起動する HIGH Phase 1
多言語対応（日英・日中） 外国籍入居者向けに英語・中国語での自動応答。入力言語を自動検出して同言語で返答 LOW Phase 3

C. エスカレーション機能（緊急案件の即時通知）

機能 詳細仕様 優先度 フェーズ
緊急キーワード検知 「水漏れ」「ガス漏れ」「鍵」「火事」「怪我」等の緊急ワードを検知した場合、AI応答とは別に即時Slack通知を発火。検知は前処理（正規表現）とAI分類の2段構え HIGH Phase 1
担当者への自動通知 Slack Webhookで物件担当者のチャンネルに通知。通知内容：入居者名・部屋番号・メッセージ本文・緊急度・AIサマリー・過去履歴リンク HIGH Phase 1
入居者への応急対応ガイド エスカレーション起動と同時に、入居者に「担当者に繋ぎました。XX分以内に連絡します。それまでの間、〇〇してください。」と送信 HIGH Phase 1
対応履歴の自動記録 エスカレーション起動から担当者が対応完了マークをつけるまでの全ログを時系列で管理DBに保存。クレーム発生時の証跡として活用 MED Phase 2

D. オーナーレポート自動生成

機能 詳細仕様 優先度 フェーズ
月次レポート自動生成 毎月1日に管理DBからデータ取得 → Claudeがオーナー向け文章を生成 → PDF化して指定メールアドレスに自動送信 HIGH Phase 3
レポート構成項目 当月の入退去状況・空室率・修繕履歴（写真付き）・問い合わせ件数・賃料収支サマリー・来月の予定（更新期日）を自動生成 HIGH Phase 3
オーナーポータル閲覧 過去12ヶ月分のレポートをWeb閲覧できるオーナー専用ページ。ログインはGoogle OAuthで認証 MED Phase 3

04 — PROMPT ENGINEERING

プロンプト設計

設計方針：物件情報・管理規約・よくある回答をシステムプロンプトに注入し、Claudeが物件ごとに正確な回答を生成できる構造を設計する。個人情報は部屋IDのみを使い、名前・連絡先はAPIに渡さない。緊急検知ルールをハードルールとして最上位に定義し、AIが安全側に倒す判断をする。
system: | あなたは【物件管理会社名】の入居者対応サポートAIです。 【最優先ルール — 緊急エスカレーション】 以下のいずれかを検知した場合、即座に escalate: true を返す: - 水漏れ・雨漏り・ガス漏れ・火災・地震被害の言及 - 鍵紛失・不法侵入・ストーカー・犯罪の言及 - 入居者が怒り・恐怖・強い不満を表現している - AIへの信頼度が低下し「人間に繋いでほしい」と要求 - 回答自信度スコアが 0.7 未満 【物件コンテキスト（動的注入）】 物件ID : {property_id} 物件名 : {property_name} 管理規約 : {property_rules_text} FAQ関連項目: {faq_retrieved_items} 入居者部屋 : {room_number} 過去7日の会話: {conversation_history} 【口調・制約】 - 敬語・丁寧語で統一。「〜でございます」は使わない - 法的・医療・金銭トラブルには具体的判断をせず「確認して折り返す」 - 規約に記載のない事項は「確認します」と答え、確定情報として返答しない - 回答は300文字以内。長くなる場合は要点をリスト化する # 出力形式（JSON Structured Output） { "category" : "emergency | move_out | renewal | equipment | noise | other", "urgency" : "critical | high | normal | low", "escalate" : true | false, "confidence" : 0.0 〜 1.0, "response" : "入居者への返答テキスト", "staff_summary": "エスカレーション時の担当者向け状況サマリー" }
RAG（検索拡張生成）設計：物件規約・FAQ・手続きガイドをChroma DBにVector化して保存。入居者のメッセージから意味的類似度の高い上位3〜5件を取得してシステムプロンプトに注入。Claudeが「提供された情報のみで回答する」ことをハードルールとして定義し、ハルシネーション（事実と異なる回答）を防止する。
