01 — PROJECT OVERVIEW

プロジェクト概要

📋
課題 01 — 勘と経験に依存した発注
ベテランシェフの頭の中にしかない発注ノウハウ。担当者が休むと発注量の根拠が消える。 属人化した知識は引き継ぎができず、廃棄と欠品の両方が発生する。
🌦
課題 02 — 天気・イベントの手動考慮
明日の天気や近隣イベントの影響を毎回手動で考慮しており、予測作業だけで1日30分かかっている。 繁忙期の判断ミスが大量廃棄に直結している。
📊
課題 03 — 廃棄の原因分析が不可能
何が余って何が足りなかったか記録が残らない。廃棄が発生しても「なぜ」が分からず改善できない。 経験則は更新されないまま同じミスを繰り返す。
解決アプローチ： POSシステムの売上データ・天気API・イベントカレンダー・廃棄ログをClaude API（claude-sonnet-4-6）が複合分析。 翌日の来客数とメニュー別需要を予測し、食材の推奨発注量を自動算出する。 シェフは承認するだけで発注が完了し、廃棄ログが蓄積されるほど予測精度が向上する自己改善ループを実現する。

03 — FUNCTIONAL REQUIREMENTS

機能要件

A. データ収集・連携機能

機能 詳細仕様 優先度 フェーズ
POSシステム連携 主要POSシステム5種（Square・Airレジ・ぐるなびセルフ等）とのAPIまたはCSV取込。日次で自動同期 HIGH Phase 1
天気API自動取得 翌日〜3日先の気温・天候を自動取得。外気温15℃以下は温かいメニュー需要増として反映 HIGH Phase 1
イベントカレンダー手動登録 近隣の大規模イベント・祝日・特別営業日をスタッフが登録。カレンダーUIで視覚管理 MED Phase 1
廃棄ログ入力 スタッフがタブレットで食材別廃棄量を入力。翌日の予測学習データとして蓄積 HIGH Phase 1

B. AI分析・予測機能

機能 詳細仕様 優先度 フェーズ
売上トレンド分析 過去30日のメニュー別販売数を分析。週次・月次のトレンドを自動検出 HIGH Phase 1
複合需要予測 曜日×天気×イベントの3要素を掛け合わせた翌日来客数と注文比率の予測。直近3週間の同曜日データを優先 HIGH Phase 1
食材展開（レシピDB連携） メニュー別の食材使用量をレシピDBから自動逆算。「鶏の唐揚げ 30食分」→「鶏むね肉 4.5kg」へ変換 HIGH Phase 1
信頼区間付き予測量表示 推奨量に加え最小・最大の幅を表示（例：「鶏むね肉: 推奨2.3kg ±0.4kg」）。シェフが安心して修正できる根拠を提供 MED Phase 1

C. 発注管理画面

機能 詳細仕様 優先度 フェーズ
AI推奨リストの確認・修正 推奨発注量をシェフが直接編集できる。修正した場合は「手動修正済み」フラグが付き学習データに反映 HIGH Phase 1
業者別発注書自動作成 承認ボタン押下で業者ごとにフォーマットした発注書を自動生成。メール送信またはFAX出力に対応 HIGH Phase 1
廃棄ロス分析ダッシュボード 週次・月次の廃棄額推移グラフ。食材別の廃棄率ランキング。AI導入前後の比較レポート MED Phase 2
アラート機能 予測来客数が通常比150%以上の日は「増量アラート」を表示。在庫ゼロリスクが高い食材に赤フラグ MED Phase 2

D. 非機能要件

項目 要件 対応方法
予測精度 需要予測の誤差±15%以内（稼働3ヶ月後の目標） 廃棄ログを蓄積するほど精度が向上する自己改善ループ。初期は安全在庫を多めに設定し、徐々に絞る
操作性 非IT系スタッフが3回で操作できること 廃棄ログ入力画面はタブレット最適化・大型タッチターゲット。ステップ数は3以内に制限
POS連携 主要POSシステム5種に対応（Square・Airレジ・ぐるなびセルフ等） API接続を優先。APIなしの場合はCSVエクスポート→自動取込フローで代替
オフライン対応 ネット障害時は前回の発注量で仮発注できること 前回発注データをローカルキャッシュ。「オフラインモード」で前回値を表示し手動承認を可能にする
セキュリティ 発注データ・売上データの保護。不正アクセス遮断 Supabase RLSによるアクセス制御。API キーはサーバーサイドのみ管理。TLS 1.3通信

05 — PROMPT ENGINEERING

AIの予測ロジック / プロンプト設計

設計方針： シェフが「なぜこの量か」を理解できるよう、推奨量だけでなく推論の根拠を必ず1文で付与する設計にする。 JSON出力を強制することで、発注管理画面側でのパース処理を安定させる。 廃棄ログを学習データとして蓄積するほど予測精度が向上する自己改善ループを構成する。
system: | あなたは飲食店の食材発注を最適化するAIです。 過去の売上データ・天気・イベント・廃棄ログを分析し、 翌日の食材発注量を正確に予測してください。 # 出力ルール - 必ずJSONのみで出力する（説明文・Markdown不要） - confidence は HIGH / MEDIUM / LOW の3段階のみ - reasoning は推奨理由を1文（40字以内）で記述 - special_notes はなければ空文字を返す --- user: | 【入力データ】 直近30日間のメニュー別販売数: {sales_history} 明日の予想天気: {weather} 近隣イベント: {events} 直近3週間の同曜日売上: {same_weekday_sales} 現在の在庫量: {current_stock} 廃棄ログ（直近2週間）: {waste_log} --- # 期待するJSON出力形式 { "predicted_covers": 予測来客数, "confidence": "HIGH | MEDIUM | LOW", "items": [ { "ingredient": "食材名", "recommended_order": 数量, "unit": "単位（kg / 個 / 缶 等）", "confidence_range": { "min": 最小量, "max": 最大量 }, "reasoning": "この量を推奨する理由（1文）" } ], "special_notes": "注意事項（明日は花見客が多い等）" }
信頼度判定ロジック： HIGH — 直近3週間の同曜日データが揃っており、天気・イベントの変動が少ない通常日。 MEDIUM — 気温変化±5℃以上または近隣イベントあり。廃棄ログが2週間未満の食材。 LOW — 連休初日・イベント直前など過去データが少ない特殊日。在庫変動の大きい食材。
